# 資料構成とストーリーボード

S2a（ストーリーボード）で使う中核リファレンス。
既定の流れは `request.yaml.presentation_type` から選ぶ。タイプが曖昧な場合は、最も近い構成を選んでその前提をユーザーに確認する。

## ストーリーボードの作り方（S2a）

スライドを 1 枚も書く前に、`.slide-work/storyboard.md` で論理を固める。

### 1. ピラミッドを立てる（結論先行）

- **頂点**: 聞き手にしてほしい判断・行動を 1 文で書く（例: 「PoC 予算 300 万円を承認してほしい」）
- **キーライン**: 頂点を支える論拠を 3 個前後（2〜4 個）挙げる。互いに重複せず、合わせると頂点を支え切る（MECE）こと
- **データ**: 各論拠を支える事実・数値・出典を 1 行ずつ添える
- 分析は下から積み上げてよいが、**提示は必ず上から**（結論 → 論拠 → データ）

### 2. 冒頭を SCQA で設計する

2 枚目（executive summary）に入れる内容を決める:

- **S（状況）**: 聞き手が同意できる現状
- **C（複雑化）**: 放置できない変化・問題
- **Q（問い）**: 聞き手の頭に浮かぶ問い
- **A（答え）**: 頂点の主張 = このデッキの結論

### 3. 全スライドのアクションタイトルを先に書く

構成パターン（下記）に沿って、各スライドのタイトルを**確定文**で書き切る。
アクションタイトルの基準:

- スライドの結論を述べる完全な文。トピックラベル（「〜について」「背景」）にしない
- 40 全角字以内・2 行以内。能動態。可能なら数字を入れる
- 「および」「かつ」で 2 主張をつなぐくらいなら 2 枚に分ける

| 悪い例（トピックラベル） | 良い例（アクションタイトル） |
|---|---|
| 在庫管理の現状について | 滞留在庫のコストは年間4,200万円に達している |
| 課題の分析 | 原因は需要予測の精度ではなく、発注の属人化にある |
| 導入スケジュール | 2026年度内に全社展開まで到達できる |

### 4. 横のロジックを検証する

タイトルだけを上から順に読み、**それだけで主張が一貫したストーリーとして通るか**確認する（reviewer G11 が同じテストをする）。通らなければタイトルを直してから先へ進む。

### 5. archetype を割り当てる

各スライドに `references/layout-patterns.md` の archetype を割り当てる。同一 archetype が 3 連続しないよう並びを調整する。

### storyboard.md の形式

```markdown
# storyboard
推奨（頂点）: <聞き手にしてほしい判断を 1 文>
論拠: 1) ... 2) ... 3) ...

| # | archetype | アクションタイトル（確定文） | 載せる根拠・内容（1 行） |
|---|-----------|------------------------------|--------------------------|
| 1 | title-hero | ... | ... |
```

## 共通の既定値

- 1 枚目はサブタイトルと対象読者/文脈の手がかりを含むタイトルスライドにする
- 2 枚目はエグゼクティブサマリーまたは主たる提案にする（タイプ別パターンに 2 枚目の指定要素がある場合は、それがエグゼクティブサマリーを兼ねる）
- 最終スライドでは判断、アクション、または次の一手を求める（担当か期限を添える）
- 7 枚以上のデッキには少なくとも 1 枚の section divider を入れる
- ナビゲーション自体が聞き手に有益でない限り、agenda スライドは省く

## 提案資料

1. タイトル
2. エグゼクティブサマリー（例: 「結論: 300万円のPoCで半減シナリオを検証させてほしい」）
3. 課題または機会（例: 「滞留在庫のコストは年間4,200万円に達している」）
4. 根拠または緊急性（例: 「原因は需要予測ではなく、発注の属人化にある」）
5. 提案する解決策（例: 「既存基幹システムに追加開発なしで接続できる」）
6. 想定される効果または ROI（例: 「投資は初年度内に回収できる見込み」）
7. 展開計画または次のアクション（例: 「本日ご判断いただきたいこと」）

## エグゼクティブ向け進捗報告

1. タイトル
2. エグゼクティブサマリー（例: 「計画比 92% で進行中、判断が1件必要」）
3. 目標に対する進捗（例: 「主要3マイルストーンのうち2つを完了した」）
4. 主要な数値（例: 「稼働率は目標の 95% に到達した」）
5. リスクまたは障害（例: 「外部接続の遅延が全体を2週間押す恐れがある」）
6. 必要な判断（例: 「追加要員2名の投入を今週中に決めたい」）
7. 次のステップ

## 報告資料

1. タイトル
2. 判明事項の要約（例: 「解約の主因は価格ではなくオンボーディングだった」）
3. 対象範囲または方法論
4. 主な発見（例: 「初週に設定を完了した顧客の継続率は 2.3 倍高い」）
5. 示唆（例: 「改善投資は営業よりサポートに向けるべきである」）
6. 提言
7. 次のアクション

## 研修資料

1. タイトル
2. 学習者ができるようになること（例: 「今日で◯◯を一人で設定できるようになる」）
3. なぜ重要か（例: 「設定ミスは障害原因の 4 割を占める」）
4. 中核概念
5. 例または手順解説
6. 演習またはチェックリスト
7. 重要な要点

## リサーチ資料

1. タイトル
2. リサーチクエスチョンと結論（例: 「◯◯は有効か → 条件付きで有効である」）
3. 背景
4. 手法
5. 根拠（例: 「効果は対照群比 +18pt、ただし2条件に限られる」）
6. 解釈
7. 提言または未解決の問い

## ガイダンス

- デッキ全体を通して主張型のスライドタイトルを使う（section-divider のみ章ラベル可）
- 詳細より先に判断を前に出す
- デッキ中盤は 課題 → 根拠 → 解決策 → 効果 の明確な流れを保つ
- 聞き手がすでにその領域を理解している場合は、根拠を圧縮する前に導入スライドを削る
- 構成パターンの枚数と `target_slide_count` が合わないときは、低優先の要素（背景・詳細）から圧縮し、足りなければ中核要素を章に分割して section-divider で区切る
