# 視覚表現とレイアウト選択

レイアウトは固定archetypeのローテーションではなく、伝える関係に合わせて選ぶ。テンプレートのクラスは出発点であり、必須の一覧ではない。

## 選択基準

| 伝える関係 | 第一候補 | 注意点 |
|---|---|---|
| 1つの重要な数値 | 大数字 + 文脈 | 単位、期間、比較対象、出典を付ける |
| 2案の差 | 対称比較 | 評価軸を揃え、都合のよい比較にしない |
| 時間変化 | 折れ線、タイムライン | 時間軸と現在位置を明示する |
| 構成・依存関係 | ノード図、層構造 | 矢印の意味と境界を明示する |
| 手順 | プロセス、番号付きフロー | 分岐や反復を直線化しすぎない |
| 分布・構成比 | 棒、ドット、100%棒 | 3D表現を避け、母数を示す |
| 概念や雰囲気 | 写真、図解、引用 | 装飾ではなくメッセージと対応させる |
| 複数の詳細 | 表、カード、read-ahead本文 | 優先順位とグルーピングを作る |

## 共通部品

外部テーマは共通して次のクラスを提供する。

- `.cover`, `.section-break`, `.closing`
- `.kicker`, `.lede`, `.source`, `.note`, `.tag`
- `.grid-2`, `.grid-3`, `.card`, `.callout`
- `.metrics`, `.metric`, `.value`, `.label`
- `.compare`, `.before`, `.after`
- `.process`, `.step`, `.timeline`, `.milestone`
- `.quote`, `.two-col`, `.visual`
- `.bar-chart`, `.bar`, `.bar-label`, `.bar-value`

必要ならテーマCSSへ新しいクラスを追加できる。ただし、単一ページの装飾ではなく、意味のある再利用部品として追加する。

## タイポグラフィと密度

- `live`: 後方から短時間で読める大きさと情報量を優先する。
- `read-ahead`: 説明を増やしてよいが、見出し、要約、詳細の階層を作る。
- `hybrid`: 本文は簡潔にし、出典や補足を小さな領域へ分離する。
- 固定の行数やbullet数だけで判定しない。PNGで改行、余白、視線移動を確認する。
- 小さい文字で詰め込まず、分割、要約、補足資料への移動を検討する。

## 図表

- 数値を図表にする前に、比較したいものと聞き手に読み取ってほしい結論を決める。
- 軸、単位、期間、母数、出典を省略しない。
- 色だけに意味を持たせず、ラベルや形でも識別できるようにする。
- 重要な系列を強調し、その他を抑える。
- データのない装飾グラフを作らない。

## 画像・SVG

- PNG、JPG、WebP、SVGを使用できる。
- 画像は `slides/assets/` に置き、外部素材は `slides/assets/SOURCES.md` に出典とライセンスを記録する。
- SVGはファイルとして配置する。複雑な図をMarkdown内の長いinline SVGとして埋め込まない。
- Mermaidは事前にSVGへ変換する。スライド表示時のJavaScript実行に依存しない。
- アイコンは内容理解に寄与するときだけ使用し、すべてのカードへ機械的に付けない。

## 避けること

- 内容と無関係なレイアウト変更で「多様性」を演出する。
- すべてをカードで囲む。
- 数字を大きくしただけで根拠や比較文脈を省く。
- 写真の上へ読めない文字を置く。
- 左右の情報量や整列が偶然に見える配置。
- proposal用のCTAや結論先行を、trainingやresearchへ機械的に流用する。

## レンダリング確認

MarkdownではなくPNGで次を確認する。

- 視線の入口が明確か
- タイトル、主役、補足の優先順位が見えるか
- 切れ、重なり、低コントラストがないか
- 図表が主張を正しく支えているか
- 発表形態に対して文字が読めるか
- デッキ全体で余白、色、書体に一貫性があるか
