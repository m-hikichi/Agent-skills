# Narrative design

Marpの実装前に、聞き手の認知の流れを設計する。元資料の章順は情報の保管順であり、説明に最適な順とは限らない。

## 1. 成功条件を固定する

次の文を具体化する。

> この聞き手が、資料を見終えた後に ______ を理解／判断／実行できれば成功である。

聞き手の知識、反論、判断基準、閲覧形態を加える。`live` はその場の理解、`read-ahead` は単独での再読、`hybrid` は画面上の要点とnotesの補完を重視する。

## 2. 根拠をID化する

数値、引用、観測、要件へ `E-001` のようなIDを付ける。最低限、出典、対象期間、単位、母数、取得条件を記録する。次を混同しない。

- 事実: 根拠から直接確認できる
- 解釈: 事実から合理的に読めるが反証可能
- 仮説: 検証前の推定
- 提案: 将来の行動

## 3. Deck thesisとspine

deck thesisは、資料全体で一番伝えたい1文。3〜5個の主要根拠だけで支える。各根拠がthesisを支えないなら削るか補足へ移す。

典型的なspine:

1. 聞き手にとっての結論または問い
2. なぜ今扱うのか
3. 何が分かったか
4. それが何を意味するか
5. 次に何を判断／検証／実行するか

資料タイプに合わない段階は省いてよい。研修を意思決定資料のCTAで終えたり、研究の限界を結論の邪魔として隠したりしない。

## 4. Action title

タイトルはページの役割に応じて使い分ける。

- 主張ページ: 「初週設定の遅れが解約差の大半と関連する」
- 方法ページ: 「契約月を揃えた1,240社で90日継続を比較した」
- 演習ページ: 「演習：次に確認する観測を1つ選ぶ」
- 章区切り: 「02 — 判断材料」

「分析結果」「課題」「まとめ」のように本文を読まないと意味が分からないタイトルを避ける。

## 5. Ghost-deck test

action titleだけを縦に並べ、次を検査する。

- タイトル列だけで論理が説明できる
- 隣接ページの関係が「だから」「しかし」「具体的には」で言える
- 同じ結論が言い換えだけで重複していない
- 結論の直前に必要な根拠がある
- 反論、限界、代替案が適切な位置にある
- 最終ページが資料の目的に沿って閉じる

飛躍があればページを足す前に、thesis、根拠、順序のどこが壊れているか直す。

## Storyboardに残すもの

- 成功条件、deck thesis、根拠方針
- action-title列とセクション遷移
- ページの役割、根拠ID、visual briefへの入口
- `live/hybrid`で話すが画面に載せない補足
