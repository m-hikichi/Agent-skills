# 変更影響チェックリスト

コード修正の前に、次の観点で `仕様影響あり` / `仕様影響なし` / `既存仕様なし` を判定する。

## 仕様影響ありとみなす変更

- API の入出力、HTTP ステータス、イベント名、CLI オプションが変わる
- UI の表示条件、入力制約、文言、遷移、権限が変わる
- バリデーション、エラー条件、リトライ条件、既定値が変わる
- 永続化データ、スキーマ、ジョブの副作用が変わる
- SLA、性能保証、セキュリティ要件、監査要件が変わる
- 実装ファイルの移動、関数名変更、クラス名変更、コンポーネント名変更がある
- テストファイルの移動、テスト名変更、`TC-XXX` の付け替えがある

## 仕様影響なしとみなせる変更

- リファクタリングのみで外部挙動が変わらない
- コメント、ログ、内部関数分割だけの変更
- テストの保守や依存更新で観測可能な挙動が変わらない

## 既存仕様なしとみなす状態

- 変更対象に対応する `requirements.md` の要件を特定できない
- 対応する `SPEC-XXX` や `FR-` / `NFR-` を特定できない
- トレーサビリティマトリクスに対応行がない
- 対応するテストIDや検証対象を現行資料から引けない

## 判定手順

1. 変更対象のファイル、関数、クラス、エンドポイント、画面、テストを列挙する。
2. `spec-config.json`、`requirements.md`、`SPEC-XXX`、トレーサビリティマトリクスを確認し、関連する `SPEC-` / `FR-` / `NFR-` / `TC-` を探す。
3. 対応する要件、仕様、テスト、マトリクスが見つからないなら、`既存仕様なし` と判定する。
4. 見つかった場合は、上の「仕様影響あり」に 1 つでも当てはまるか確認する。
5. 当てはまるなら、`あり` と判定し、要件書、仕様書、テスト、マトリクスの更新対象を列挙する。
6. 当てはまらないなら、`なし` と判定し、「影響なし」の理由を 1 文で残す。

## 返答テンプレート

```markdown
## 仕様影響判定

- 判定: あり
- 対象要件: FC-04-04-001, NF-003
- 対象仕様: SPEC-004, FR-002, NFR-003
- 対象テスト: TC-009, TC-010
- 対象マトリクス: `docs/traceability-matrix.md` の FC-04-04-001 / SPEC-004 行
- 理由: 入力バリデーションと API エラーレスポンスが変わり、`createOrder` 関数名も `createPurchase` に変わるため
- 必要な更新:
  - `requirements.md` の対応表と受け入れ基準
  - 仕様書の `機能要件`、`実装トレーサビリティ契約`、`API仕様`
  - テストケース TC-009, TC-010
  - トレーサビリティマトリクスの対象行
```

```markdown
## 仕様影響判定

- 判定: なし
- 対象要件: FC-04-04-001
- 対象仕様: SPEC-004, FR-002
- 対象テスト: TC-009
- 対象マトリクス: `docs/traceability-matrix.md` の FC-04-04-001 / SPEC-004 行
- 理由: ログ整形と内部関数分割のみで、外部挙動とトレーサビリティ契約は変わらないため
- 必要な更新:
  - なし
```

```markdown
## 仕様影響判定

- 判定: 既存仕様なし
- 対象要件: 該当なし
- 対象仕様: 該当なし
- 対象テスト: 該当なし
- 対象マトリクス: 該当なし
- 理由: 変更対象に対応する `requirements.md`、`SPEC-XXX`、テストID、マトリクス行を特定できなかったため
- 必要な更新:
  - 対応する要件、仕様、テスト、マトリクスの新規整備が必要
```
