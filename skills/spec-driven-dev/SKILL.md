# 仕様駆動開発（Spec-Driven Development）スキル

仕様書を「信頼できる唯一の情報源（Single Source of Truth）」として開発を進めるスキル。
仕様書とコードの乖離を防ぎ、将来の自分やAIが迷わずコードを修正・拡張できる状態を維持する。

## なぜ仕様駆動開発が必要か

コードだけで開発を進めると、時間が経つにつれて「なぜこう実装したのか」「どこにどの機能があるのか」が分からなくなる。仕様書があっても更新されなければ嘘の情報源になってしまう。このスキルは、仕様書の作成から更新までのワークフローを強制的に組み込むことで、仕様とコードが常に一致した状態を保つ。

## プロジェクト設定

初回起動時、プロジェクトルートに `spec-config.json` が存在するか確認する。なければ以下のテンプレートで作成を提案する：

```json
{
  "project_name": "プロジェクト名",
  "language": "TypeScript",
  "framework": "Next.js",
  "spec_dir": "docs/specs",
  "traceability_path": "docs/traceability-matrix.md",
  "spec_id_prefix": "SPEC",
  "coding_conventions": {
    "naming": "camelCase",
    "test_framework": "Jest",
    "directory_structure": "src/ 配下に機能ごとのディレクトリ"
  }
}
```

`language` と `framework` はプロジェクトごとに自由に設定できる。スキルはこの設定に基づいてコード生成やチェックの方法を適応させる。

## ディレクトリ構成

```
project-root/
├── spec-config.json          # プロジェクト設定
├── docs/
│   ├── specs/                # 仕様書（機能単位で1ファイル）
│   │   ├── SPEC-001-user-auth.md
│   │   ├── SPEC-002-product-catalog.md
│   │   └── ...
│   ├── traceability-matrix.md  # トレーサビリティマトリクス
│   └── requirements.md         # 要件定義（必要に応じて）
└── src/                       # 実装コード
```

## 仕様書フォーマット

各仕様書は以下の構造に従う。テンプレートの詳細は `references/spec-template.md` を参照。

```markdown
---
spec_id: SPEC-001
title: ユーザー認証
status: draft | review | approved | implemented | deprecated
created: 2026-01-15
updated: 2026-01-20
author: 作成者名
---

# SPEC-001: ユーザー認証

## 概要
この機能が何をするか、なぜ必要かを簡潔に説明する。

## 機能要件
- [ ] FR-001: メールアドレスとパスワードでログインできる
- [ ] FR-002: パスワードリセット機能がある
- ...

## 非機能要件
- [ ] NFR-001: レスポンスタイム500ms以内
- ...

## データモデル
関連するデータ構造やスキーマの定義。

## API仕様（該当する場合）
エンドポイント、リクエスト/レスポンス形式。

## 画面仕様（該当する場合）
画面構成、ユーザーインタラクション。

## 判断記録
大きな方針変更があった場合のみ記録する。日常的な変更履歴はGitのコミット履歴で追跡する。

| 日付 | 判断内容 | 理由 |
|------|---------|------|
| 2026-01-15 | 認証方式にJWTを採用 | ステートレスでスケーラブルなため |
```

## トレーサビリティマトリクス

仕様IDと実装コードの対応表。これが「どこに何があるか」を一発で確認できる地図になる。

```markdown
# トレーサビリティマトリクス

## 凡例
- ✅ 実装済み・整合性確認済み
- 🔧 実装中
- ⚠️ 仕様変更あり・コード未反映
- ❌ 未実装

| 仕様ID | 要件ID | 実装ファイル | 関数/クラス | テスト | ステータス |
|--------|--------|------------|-----------|--------|----------|
| SPEC-001 | FR-001 | src/auth/login.ts | handleLogin() | tests/auth/login.test.ts | ✅ |
| SPEC-001 | FR-002 | src/auth/reset.ts | resetPassword() | tests/auth/reset.test.ts | 🔧 |
| SPEC-002 | FR-001 | src/products/list.ts | getProducts() | tests/products/list.test.ts | ✅ |
```

トレーサビリティマトリクスは仕様書やコードを変更するたびに必ず更新する。これを怠ると仕様駆動開発の意味がなくなる。

## ワークフロー

### Phase 1: 要件定義と機能一覧の洗い出し

ユーザーから「〇〇を作りたい」という依頼があったら：

1. **要件のヒアリング** — ユーザーの目的、対象ユーザー、主要機能を確認する
2. **既存コードの調査** — プロジェクトが既に存在する場合、コードベースを調査して既存の機能や構造を把握する
3. **機能一覧の作成** — 必要な機能を洗い出してリスト化する。各機能にIDを振る
4. **仕様書の分割計画** — 機能の粒度を決め、どの機能をどの仕様書にまとめるか決める
5. **ユーザーに確認** — 機能一覧と分割計画を提示し、過不足がないか確認する

### Phase 2: 仕様書の作成

1. `spec-config.json` の設定を確認する（なければ作成を提案）
2. 仕様書テンプレートに沿って `docs/specs/SPEC-XXX-機能名.md` を作成する
3. 機能要件はチェックリスト形式（`- [ ] FR-XXX:`）で書く。これが後で実装の追跡に使える
4. トレーサビリティマトリクスに新しい仕様のエントリを追加する（ステータスは ❌）
5. ユーザーにレビューを依頼する

### Phase 3: コード生成

仕様が承認されたら：

1. 仕様書を読み込み、機能要件を確認する
2. `spec-config.json` の言語・フレームワーク設定に基づいてコードを生成する
3. 生成したコードのファイルパスと関数名をトレーサビリティマトリクスに記録する
4. テストコードも生成し、マトリクスに記録する
5. 仕様書のステータスを `implemented` に更新する
6. 仕様書の機能要件チェックリストを更新する（実装した項目にチェック）

コード生成時、各ファイルの先頭にどの仕様に対応しているかをコメントで記載する：
```typescript
/**
 * @spec SPEC-001
 * @requirement FR-001, FR-002
 */
```

### Phase 4: テスト仕様の作成

コードを実装したら（または実装と並行して）、仕様書の機能要件からテストケースを導出する。テストを書く前にまず「何をテストするか」を明確にすることで、テストの抜け漏れを防ぐ。

1. **テストケースの導出** — 仕様書の各機能要件（FR-XXX）から、テストすべきケースを洗い出す
   - 正常系: 期待通りの入力で期待通りの結果が得られるか
   - 異常系: 不正な入力やエラー条件で適切に処理されるか
   - 境界値: 上限・下限・空値などの境界条件

2. **テスト一覧の作成** — 仕様書にテスト仕様セクションを追加する

```markdown
## テスト仕様

| テストID | 対応要件 | テスト内容 | 種別 |
|---------|---------|-----------|------|
| TC-001 | FR-001 | 正しいメールとパスワードでログインできる | 正常系 |
| TC-002 | FR-001 | 存在しないメールでログインするとエラー | 異常系 |
| TC-003 | FR-001 | パスワードが不正だとエラー | 異常系 |
| TC-004 | FR-002 | 新規メールで登録できる | 正常系 |
| TC-005 | FR-002 | 既存メールで登録するとエラー | 異常系 |
```

3. **ユーザーに確認** — テスト一覧を提示し、過不足がないか確認する
4. **テストコードの実装** — テスト一覧に基づいてテストコードを書く
5. **トレーサビリティマトリクスの更新** — テストファイルパスをマトリクスに記録する

### Phase 5: 整合性チェック

仕様とコードの整合性を確認する。以下を順番にチェックする：

1. **トレーサビリティマトリクスの完全性**
   - 全ての仕様IDに対応する実装ファイルがあるか
   - マトリクスに記載されたファイルパスが実際に存在するか
   - テストファイルが存在するか

2. **仕様と実装の一致**
   - 仕様書の機能要件（FR-XXX）が全て実装されているか
   - 実装が仕様書に記載のない機能を含んでいないか（スコープクリープの検出）
   - APIのエンドポイント、パラメータ、レスポンス形式が仕様と一致しているか

3. **ステータスの正確性**
   - 仕様書のステータスが実態と合っているか
   - マトリクスのステータスが最新か

不整合が見つかった場合は、具体的な差分と修正提案を提示する。

### Phase 6: コード修正時の仕様影響チェック

バグ修正や機能改善などでコードを変更する場合、その変更が仕様に影響するかを確認する。仕様書は「コードがどう振る舞うべきか」の定義なので、振る舞いが変わるならば仕様も更新する必要がある。

1. **変更内容の確認** — 何をどう変更するかを把握する
2. **仕様への影響判断** — トレーサビリティマトリクスを参照し、変更対象のファイル/関数がどの仕様ID・要件IDに紐づいているか確認する
3. **影響がある場合**:
   - 仕様書に新しい要件を追加する（例: 「NFR-003: nullの場合はエラーを返す」）
   - または既存の要件の説明を修正する
   - テスト仕様にも対応するテストケースを追加する
   - トレーサビリティマトリクスを更新する
4. **影響がない場合**:
   - 仕様書の更新は不要（内部リファクタリングなど）
   - ただし「影響なし」の判断をユーザーに伝え、確認する

判断の目安: **ユーザーから見える振る舞いが変わるなら仕様に影響あり**。内部実装の改善だけなら影響なし。迷ったらユーザーに確認する。

### Phase 7: 仕様変更

仕様変更の依頼があったら：

1. **影響範囲の特定**
   - トレーサビリティマトリクスを参照し、変更対象の仕様IDに紐づく全てのファイルを特定する
   - 変更が他の仕様に波及しないか確認する（依存関係の確認）
   - 影響を受けるファイルの一覧をユーザーに提示する

2. **仕様書の更新**
   - 仕様書の該当箇所を修正する
   - 大きな方針変更の場合は「判断記録」に理由を記録する
   - `updated` の日付を更新する
   - ステータスを `review` に戻す

3. **レビュー**
   - 変更後の仕様書をユーザーに提示してレビューを依頼する

4. **コードの修正**
   - 仕様変更が承認されたら、影響範囲のコードを修正する
   - トレーサビリティマトリクスを更新する（変更中は ⚠️ → 完了後 ✅）

5. **再整合性チェック**
   - Phase 5 の手順で仕様とコードの整合性を再確認する

## コマンドリファレンス

ユーザーが以下のような依頼をしたとき、対応するPhaseを実行する：

| ユーザーの依頼（例） | 対応するPhase |
|-------------------|-------------|
| 「〇〇を作りたい」「要件を整理して」 | Phase 1 → 2 |
| 「仕様書を作って」「スペックを書いて」 | Phase 2 |
| 「仕様からコードを生成して」「実装して」 | Phase 3 |
| 「テスト一覧を作って」「テスト仕様を書いて」 | Phase 4 |
| 「仕様とコードが合ってるかチェックして」 | Phase 5 |
| 「このバグを直して」「nullチェックを追加して」 | Phase 6（修正後に仕様影響を確認） |
| 「この仕様を変更したい」「機能を追加したい」 | Phase 7 |
| 「影響範囲を調べて」 | Phase 7 の手順1 |
| 「トレーサビリティを確認して」 | トレーサビリティマトリクスを表示 |

## 運用上のルール

1. **コードを変更したら仕様書も更新する** — コード変更時に対応する仕様書の確認と更新を必ず提案する。これがこのスキルの最も重要なルール
2. **トレーサビリティマトリクスは常に最新に保つ** — ファイルの追加・削除・リネームがあったら必ず反映する
3. **仕様書のステータスを正確に維持する** — draft → review → approved → implemented の流れを守る
4. **重要な判断は記録する** — 大きな方針変更とその理由は仕様書の「判断記録」に残す。日常的な変更履歴はGitで追跡する
5. **既存コードに仕様を後付けできる** — 新規開発だけでなく、既存プロジェクトに後から仕様書を整備することも可能。コードを読んで仕様書を逆生成し、トレーサビリティマトリクスを構築する
