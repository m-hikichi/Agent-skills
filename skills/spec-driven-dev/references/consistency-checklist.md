# 整合性チェックリスト

仕様とコードが合っているか確認するときは、次の順で見る。

## 1. requirements.md の完全性

- `プロダクトゴール`、`対象ユーザー`、`MVPスコープ`、`MVP対象外`、`受け入れ基準` があるか
- `FC-` / `NF-` の採番規則が崩れていないか
- すべての主要要件に `対応仕様` があるか

## 2. 仕様書の完全性

- 対象仕様に `spec_id`、タイトル、ステータス、更新日があるか
- `対応要件（requirements.md）` 表があるか
- FR/NFR の ID が重複していないか
- 依存仕様や前提条件が明記されているか
- 実装済みと主張しているのに、未解決事項が放置されていないか

## 3. マトリクスの完全性

- 各 `FC-` / `NF-` に少なくとも 1 行の対応があるか
- マトリクス上のファイルパスが実在するか
- 主要シンボル名が現実のコードと一致するか
- テストファイルやテストケースが存在するか

## 4. 実装との一致

- 仕様に書かれた振る舞いがコードとテストで観測できるか
- 仕様に無い API、画面遷移、入力制約、エラー条件が増えていないか
- データモデルやレスポンス形式が仕様どおりか
- ステータスやコメントが現状に追従しているか

## 5. レポートの返し方

不整合は次の形で返す。

```markdown
## Findings

1. High: SPEC-001 / FR-003 が未実装
   - 根拠: マトリクスに `❌`、対応コードなし
   - 影響: 仕様では必須機能だがリリース対象に入らない
   - 修正案: 実装するか、仕様の優先度とステータスを見直す
```

重大度は `High`、`Medium`、`Low` の 3 段階で十分。
