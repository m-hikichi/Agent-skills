# 整合性チェックリスト

コードまたは仕様書を変更したら、このチェックを必ず最後に実行する。差分があれば報告だけで終わらず、修正してからもう一度このチェックに戻る。

## 0. 実行タイミングと完了条件

- コード変更後は必ず実行する
- 仕様書変更後も必ず実行する
- `requirements.md`、`SPEC-XXX`、コード、テスト、マトリクスのうち変更に関係するものをすべて対象にする
- 完了を主張する前に `scripts/verify_spec_consistency.py` を成功させる
- 完了を主張する前に `project_test_commands` を成功させる
- `High` / `Medium` の不整合が 0 件になり、残る `Low` も許容理由を説明できるまで完了扱いにしない
- 最終メッセージの末尾に `## 最終整合性監査` を入れる

そのうえで、次の順で見る。

## 1. requirements.md の完全性

- `プロダクトゴール`、`対象ユーザー`、`MVPスコープ`、`MVP対象外`、`受け入れ基準` があるか
- `FC-` / `NF-` の採番規則が崩れていないか
- すべての主要要件に `対応仕様` があるか

## 2. 仕様書の完全性

- 対象仕様に `spec_id`、タイトル、ステータス、更新日があるか
- `対応要件（requirements.md）` 表があるか
- FR/NFR の ID が重複していないか
- `実装トレーサビリティ契約` があり、すべての `FR-` / `NFR-` をカバーしているか
- `実装完了条件` があり、すべての `FR-` / `NFR-` に観測可能な結果があるか
- 依存仕様や前提条件が明記されているか
- 実装済みと主張しているのに、未解決事項が放置されていないか

## 3. マトリクスの完全性

- 各 `FC-` / `NF-` に少なくとも 1 行の対応があるか
- マトリクス上のファイルパスが実在するか
- `実装トレーサビリティ契約` とマトリクスのシンボル種別、シンボル名、テストIDが一致するか
- テストファイルやテストケースが存在するか
- テストコードから `TC-XXX` を検索できるか

## 4. 実装との一致

- 仕様に書かれた振る舞いがコードとテストで観測できるか
- 仕様に無い API、画面遷移、入力制約、エラー条件が増えていないか
- データモデルやレスポンス形式が仕様どおりか
- 仕様書に書かれたファイル名、関数名、クラス名が現実のコードと一致するか
- ステータスやコメントが現状に追従しているか

## 5. 差分が見つかったときの修正ループ

1. どの要件ID、仕様ID、コード、テストがずれているかを特定する。
2. `approved` の仕様とユーザー確認済みの判断を優先し、どちらを直すべきか決める。
3. コード、仕様書、要件書、テスト、マトリクスの必要箇所を修正する。
4. 同じ対象範囲でもう一度このチェックリストを回す。
5. 差分が残っていれば 1 に戻る。
6. 外部仕様の未確定だけは `needs-user-decision` として止めてよいが、その場合は何が未確定かを明示してユーザー確認を求める。

## 6. レポートの返し方

不整合は次の形で返す。

```markdown
## Findings

1. High: SPEC-001 / FR-003 が未実装
   - 根拠: マトリクスに `❌`、対応コードなし
   - 影響: 仕様では必須機能だがリリース対象に入らない
   - 修正案: 実装するか、仕様の優先度とステータスを見直す
```

重大度は `High`、`Medium`、`Low` の 3 段階で十分。最終的には [references/final-audit-template.md](final-audit-template.md) に沿って、`clean` か `needs-user-decision` かを必ず明示する。
