# 既存コードから仕様を起こす手順

既存コードに仕様書が無いまま変更依頼が来たときは、全面的なドキュメント化ではなく「今回の判断に必要な最低限の仕様化」を行う。

## 使う材料

- 実装コード
- 既存テスト
- README や設計メモ
- API スキーマ、OpenAPI、画面定義
- issue やコミットメッセージ

## 手順

1. 変更対象のエントリポイントを特定する。
2. ユーザーから見える振る舞いを列挙する。
3. 先に `requirements.md` の最小版を作り、`FC-` / `NF-` と受け入れ基準を置く。
4. 振る舞いを機能単位にまとめ、必要な数だけ `SPEC-` を切る。
5. 観測できた事実だけを `FR-` / `NFR-` に落とし込む。
6. `実装トレーサビリティ契約` と `実装完了条件` を、観測できた実装ファイル、シンボル名、テストIDだけで埋める。
7. 推測が混ざる箇所は `Open Questions` や `未確定事項` として分離する。
8. `requirements.md` の対応表とトレーサビリティマトリクスを作り、今回触るファイルを結ぶ。
9. 変更後に仕様との差分を再度確認し、必要なら仕様を正とする形で整える。

## 逆生成時の注意

- 実装の癖をそのまま正当化しない。観測事実と望ましい仕様を分ける。
- テストがあるなら、期待されている振る舞いの一次資料として優先する。
- ファイル名、関数名、クラス名、テストIDは、本文だけでなく `実装トレーサビリティ契約` にも必ず残す。
- 既存挙動がバグか仕様か曖昧なら、決め打ちせずユーザーに確認する。
- 今回の変更と無関係な領域まで広げすぎない。
