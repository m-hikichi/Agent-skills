# 仕様書テンプレート

このテンプレートを使って仕様書を作成する。プロジェクトの性質に応じてセクションの追加・省略は可能だが、フロントマターと「概要」「機能要件」は必須。

---

## フルテンプレート

```markdown
---
spec_id: SPEC-XXX
title: 機能名
status: draft
created: YYYY-MM-DD
updated: YYYY-MM-DD
author: 作成者名
related_specs: []
---

# SPEC-XXX: 機能名

## 概要

この機能の目的と背景を2〜3文で説明する。
- 誰が使うのか
- 何を解決するのか
- なぜ必要なのか

## 機能要件

実装すべき機能をチェックリスト形式で列挙する。各要件にはIDを振る。

- [ ] FR-001: 要件の説明
- [ ] FR-002: 要件の説明
- [ ] FR-003: 要件の説明

### 優先度

| 要件ID | 優先度 | 備考 |
|--------|--------|------|
| FR-001 | 必須 | MVP |
| FR-002 | 必須 | MVP |
| FR-003 | あれば良い | v2以降 |

## 非機能要件

パフォーマンス、セキュリティ、可用性などの要件。

- [ ] NFR-001: レスポンスタイム500ms以内
- [ ] NFR-002: 入力値のバリデーション実施

## データモデル

関連するデータ構造。テーブル定義、型定義など。

### 例: ユーザーモデル

| フィールド | 型 | 必須 | 説明 |
|-----------|-----|------|------|
| id | string (UUID) | ✅ | 一意識別子 |
| email | string | ✅ | メールアドレス |
| name | string | ✅ | 表示名 |
| created_at | datetime | ✅ | 作成日時 |

## API仕様

RESTful APIのエンドポイント定義。

### POST /api/resource

**リクエスト:**
```json
{
  "field1": "value1",
  "field2": "value2"
}
```

**レスポンス（200 OK）:**
```json
{
  "id": "uuid",
  "field1": "value1",
  "created_at": "2026-01-15T10:30:00Z"
}
```

**エラーレスポンス:**
| コード | 説明 |
|--------|------|
| 400 | バリデーションエラー |
| 401 | 認証エラー |
| 404 | リソースが見つからない |

## 画面仕様

画面の構成要素とユーザーインタラクション。

### 画面構成
- ヘッダー: ナビゲーション、ユーザーメニュー
- メインエリア: コンテンツ表示
- フッター: コピーライト

### ユーザーフロー
1. ユーザーが画面にアクセスする
2. フォームに入力する
3. 送信ボタンを押す
4. 結果が表示される

## テスト仕様

機能要件（FR-XXX）から導出したテストケースの一覧。テストコードを書く前にまずここで「何をテストするか」を定義する。

| テストID | 対応要件 | テスト内容 | 種別 |
|---------|---------|-----------|------|
| TC-001 | FR-001 | 正常な入力で期待通りの結果が得られる | 正常系 |
| TC-002 | FR-001 | 不正な入力でエラーが返される | 異常系 |
| TC-003 | FR-001 | 境界値（空文字、最大長など）で適切に処理される | 境界値 |

## 依存関係

この機能が依存する他の仕様や外部サービス。

| 依存先 | 種別 | 説明 |
|--------|------|------|
| SPEC-001 | 内部 | ユーザー認証が必要 |
| Stripe API | 外部 | 決済処理 |

## 判断記録
大きな方針変更があった場合のみ記録する。日常的な変更はGitのコミット履歴で追跡する。

| 日付 | 判断内容 | 理由 |
|------|---------|------|
| YYYY-MM-DD | 例: 認証方式にJWTを採用 | ステートレスでスケーラブルなため |
```
