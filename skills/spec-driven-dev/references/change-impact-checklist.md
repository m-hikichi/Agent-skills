# 変更影響チェックリスト

コード修正の前に、次の観点で「仕様影響あり / なし」を判定する。

## 仕様影響ありとみなす変更

- API の入出力、HTTP ステータス、イベント名、CLI オプションが変わる
- UI の表示条件、入力制約、文言、遷移、権限が変わる
- バリデーション、エラー条件、リトライ条件、既定値が変わる
- 永続化データ、スキーマ、ジョブの副作用が変わる
- SLA、性能保証、セキュリティ要件、監査要件が変わる
- 実装ファイルの移動、関数名変更、クラス名変更、コンポーネント名変更がある
- テストファイルの移動、テスト名変更、`TC-XXX` の付け替えがある

## 仕様影響なしとみなせる変更

- リファクタリングのみで外部挙動が変わらない
- コメント、ログ、内部関数分割だけの変更
- テストの保守や依存更新で観測可能な挙動が変わらない

## 判定手順

1. 変更対象のファイル、関数、クラス、エンドポイント、画面、テストを列挙する。
2. トレーサビリティマトリクスから関連する `SPEC-` / `FR-` / `NFR-` / `TC-` を引く。
3. `requirements.md` の `FC-` / `NF-`、MVPスコープ、受け入れ基準まで変わるか確認する。
4. 上の「仕様影響あり」に 1 つでも当てはまるか確認する。
5. 当てはまるなら、要件書、仕様書、`実装トレーサビリティ契約`、`実装完了条件`、テスト仕様、マトリクスの更新内容を決める。
6. 当てはまらないなら、「影響なし」の理由を 1 文で残す。
7. そもそも対応する仕様が無いなら、先に逆生成する。

## 返答テンプレート

```markdown
## 仕様影響判定

- 判定: あり
- 対象要件: FC-04-04-001, NF-003
- 対象仕様: SPEC-004, FR-002, NFR-003
- 理由: 入力バリデーションと API エラーレスポンスが変わり、`createOrder` 関数名も `createPurchase` に変わるため
- 必要な更新:
  - `requirements.md` の対応表と受け入れ基準を更新
  - 仕様書の `機能要件`、`実装トレーサビリティ契約`、`API仕様` を更新
  - テストケース TC-009, TC-010 を追加
  - マトリクスの対象行を `⚠️` -> `✅` に更新
```
