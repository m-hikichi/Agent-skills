# 仕様書テンプレート

このテンプレートを使って仕様書を作成する。`requirements.md` で WHAT を定義した後、その要件を具体化する設計資料として使う。

このテンプレートでは、本文の説明に加えて `実装トレーサビリティ契約` と `実装完了条件` を持たせる。これにより、どこに実装されるか、どのテストで担保するか、何をもって完了とみなすかを後から機械検証しやすくする。

## フルテンプレート

````markdown
---
spec_id: SPEC-XXX
title: 機能名
status: draft
created: YYYY-MM-DD
updated: YYYY-MM-DD
author: 作成者名
related_specs: []
---

# SPEC-XXX: 機能名

## 概要

この機能の目的と背景を2〜3文で説明する。

## 対応要件（requirements.md）

| 要件ID | 要件概要 | 本仕様での実装機能ID |
|--------|----------|----------------------|
| FC-01-01-001 | requirements.md 側の要件概要 | FR-001, FR-002 |
| NF-001 | requirements.md 側の非機能要件概要 | NFR-001 |

## 機能仕様（要件実装一覧）

### グループA: 機能グループ名
- [ ] FR-001: 要件の説明
- [ ] FR-002: 要件の説明

### グループB: 機能グループ名
- [ ] FR-003: 要件の説明

### 優先度

| 要件 | 優先度 | 備考 |
|------|--------|------|
| FR-001 | 必須 | MVP |
| FR-002 | 必須 | MVP |
| FR-003 | 推奨 | UX 向上 |

## 実装トレーサビリティ契約

自由文ではなく、この表で「どこに実装されるか」を固定する。1 行は 1 つの `FR-` / `NFR-` と 1 つの実装シンボルの対応にする。同じ機能が複数シンボルにまたがるなら、機能IDを重複させて行を分ける。

| 機能ID | 実装ファイル | シンボル種別 | シンボル名 | テストファイル | テストID | 備考 |
|--------|--------------|--------------|------------|----------------|----------|------|
| FR-001 | src/todos/create.ts | function | createTodo | tests/todos/create.test.ts | TC-001 | 作成処理の入口 |
| FR-001 | src/todos/CreateTodoForm.tsx | component | CreateTodoForm | tests/todos/CreateTodoForm.test.tsx | TC-002 | 作成画面の入力 |
| FR-002 | src/todos/CreateTodoForm.tsx | component | CreateTodoForm | tests/todos/CreateTodoForm.test.tsx | TC-003 | 画面入力 |
| NFR-001 | src/todos/create.ts | function | createTodo | tests/perf/todos.test.ts | TC-101 | 200ms 以内 |

## 実装完了条件

各 `FR-` / `NFR-` について、外から観測できる完了条件を書く。曖昧な言い回しではなく、テストで判定できる粒度にする。

| 機能ID | 観測可能な結果 | テストID | 自動化 | 備考 |
|--------|----------------|----------|--------|------|
| FR-001 | 正常な入力で Todo が作成され、レスポンスに生成 ID が含まれる | TC-001 | yes | API テスト |
| FR-001 | 作成画面から正常に Todo を登録できる | TC-002 | yes | UI テスト |
| FR-002 | 必須項目が空のとき送信ボタンが無効化される | TC-003 | yes | UI テスト |
| NFR-001 | 100 件投入時でも P95 が 200ms 未満である | TC-101 | yes | 性能測定 |

## 非機能要件

パフォーマンス、セキュリティ、可用性などの要件。

- [ ] NFR-001: 非機能要件の説明
- [ ] NFR-002: 非機能要件の説明

## API仕様

RESTful APIのエンドポイント定義。

### POST /api/resource

**リクエスト:**
```json
{
  "field1": "value1",
  "field2": "value2"
}
```

**レスポンス（200 OK）:**
```json
{
  "id": "uuid",
  "field1": "value1",
  "created_at": "2026-01-15T10:30:00Z"
}
```

**エラーレスポンス:**
| コード | 説明 |
|--------|------|
| 400 | バリデーションエラー |
| 401 | 認証エラー |
| 404 | リソースが見つからない |
````

## 運用メモ

- `related_specs` には直接依存する `SPEC-` を入れる。
- `対応要件（requirements.md）` 表は省略しない。
- `requirements.md` の `FC-` / `NF-` と、この仕様の `FR-` / `NFR-` が追えるようにする。
- 既存IDを再利用しない。削除ではなく `deprecated` を使う。
- ファイル名、関数名、クラス名、テストファイル、テストIDは本文に散らさず、必ず `実装トレーサビリティ契約` にも書く。
- `実装完了条件` は「適切に処理する」のような曖昧表現を避け、観測可能な結果で書く。
- テストコード側にも `TC-XXX` を検索できる文字列を残す。
- 1 つの仕様書に複数の画面や API が入ってもよいが、1 つの責務にまとまる粒度を守る。
- 実装前に不明点が残るなら本文に曖昧なまま埋め込まず、レビュー待ちの事項としてユーザーに確認する。
