# Git Flow Commit スキル

Git Flowモデルに基づいたブランチ運用とコミットワークフローを標準化するスキル。
コミットの品質を保ち、チーム開発をスムーズに進めるためのルールを適用する。

## ブランチ戦略

Git Flowをベースとした5種類のブランチを使い分ける。

### 永続ブランチ（常に存在する）

| ブランチ | 役割 | 直接コミット |
|---------|------|------------|
| `main` | 本番環境。リリースタグを付与 | 禁止 |
| `develop` | 開発のベース。次回リリースの統合先 | 禁止 |

### 一時ブランチ（目的が終わったら削除）

| ブランチ | 命名規則 | 分岐元 | マージ先 |
|---------|---------|--------|---------|
| feature | `feature/<イシュー番号>-<簡潔な説明>` | develop | develop |
| release | `release/<バージョン>` | develop | main, develop |
| hotfix | `hotfix/<イシュー番号>-<簡潔な説明>` | main | main, develop |

### ブランチ命名の例

```
feature/42-user-login
feature/108-api-rate-limit
release/1.2.0
hotfix/55-null-pointer-fix
```

ブランチ名は英語の小文字とハイフンを使い、短く要点を表す。イシュー番号がある場合は必ず含める。

## コミットメッセージ

### フォーマット

```
<prefix>: #<イシュー番号> <変更内容の要約（日本語）>

<なぜこの変更が必要だったかの説明（任意）>
```

イシュー番号がない場合は省略してよい：
```
<prefix>: <変更内容の要約（日本語）>
```

### prefix一覧

| prefix | 用途 | 例 |
|--------|------|-----|
| `feat` | 新機能の追加 | `feat: #42 ユーザー認証機能を追加` |
| `fix` | バグ修正 | `fix: #55 ログイン時のNullPointerExceptionを修正` |
| `docs` | ドキュメントのみの変更 | `docs: #70 APIエンドポイントの説明を更新` |
| `refactor` | 機能変更を伴わないコード改善 | `refactor: #80 認証ロジックをサービス層に分離` |
| `test` | テストの追加・修正 | `test: #42 ログイン機能のユニットテストを追加` |
| `chore` | ビルド・依存関係・設定の変更 | `chore: #90 Node.jsを18にアップグレード` |
| `style` | フォーマット変更（動作に影響しない） | `style: インデントをスペース2つに統一` |
| `perf` | パフォーマンス改善 | `perf: #100 クエリのN+1問題を解消` |
| `remove` | コード・ファイルの削除 | `remove: 使用されていないヘルパー関数を削除` |
| `rename` | ファイル・関数のリネーム | `rename: UserService → AuthServiceに名称変更` |

### コミットメッセージのルール

1. **prefixは必須**。変更の性質に最も合うものを1つ選ぶ
2. **要約は日本語で書く**。簡潔に「何をしたか」を伝える
3. **イシュー番号がある場合はprefixの直後に付与**する（`prefix: #123 要約` の形式）
4. 本文（2行目以降）では「なぜこの変更が必要だったか」を説明する。1行目だけで意図が明確な場合は省略してよい
5. 1つのコミットには1つの論理的変更だけを含める。機能追加とバグ修正を同じコミットに混ぜない

## 保護ブランチへの警告

コミット前に現在のブランチを確認し、以下のブランチにいる場合は**必ずユーザーに警告**する：

- `main` — 本番ブランチ。直接コミットは禁止
- `develop` — 開発統合ブランチ。直接コミットは禁止
- `release/*` — リリース準備中。緊急修正以外のコミットは避けるべき

警告を出した上で、適切なfeatureブランチまたはhotfixブランチの作成を提案する。ユーザーが明確に「それでもこのブランチにコミットしたい」と言った場合のみ続行する。

## コミット前の安全チェック

コミット前に以下のチェックを順番に実施する。問題が見つかった場合はコミットを中断し、ユーザーに報告する。
チェック項目は今後追加される可能性がある。

### 機密情報の検出

ステージング済みの変更内容（`git diff --cached`）に機密情報が含まれていないかチェックする。以下のパターンに該当するものが見つかった場合は**必ず警告し、コミットを中断**する。一度コミットするとGitの履歴に残り、完全な削除が非常に困難になるため、水際で防ぐことが重要。

**チェック対象ファイル：**
- `.env`、`.env.local`、`.env.production` など環境変数ファイル
- `credentials.json`、`serviceAccount.json` などの認証ファイル
- `id_rsa`、`*.pem`、`*.key` などの秘密鍵ファイル

**チェック対象パターン（diffの中身）：**
- APIキー・トークン：`API_KEY=`、`SECRET_KEY=`、`ACCESS_TOKEN=`、`Bearer ` などのパターン
- パスワード：`password=`、`passwd=`、`DB_PASSWORD=` などのパターン
- AWS認証情報：`AKIA`で始まる文字列（AWSアクセスキーID）
- 秘密鍵：`-----BEGIN (RSA |EC |OPENSSH )?PRIVATE KEY-----`

検出した場合の対応：
1. 該当ファイルとパターンをユーザーに提示する
2. `.gitignore`への追加を提案する
3. 環境変数や秘密管理ツール（例：`.env.example`をテンプレートとして管理）の利用を勧める
4. ユーザーが「問題ない」と明確に判断した場合のみ続行する

### 大きなファイルの警告

コミット対象のファイルにサイズが大きいものが含まれていないか確認する。ステージングされたファイル（または `git status` で変更が検出されたファイル）に対して、以下のコマンドでサイズをチェックする：

```bash
git diff --cached --name-only | xargs ls -la
```

以下の基準で警告を出す：

| サイズ | 対応 |
|-------|------|
| 5MB以上 | **警告**：Git管理に適さない可能性がある。Git LFSの利用や`.gitignore`への追加を提案する |
| 50MB以上 | **強い警告**：GitHubのファイルサイズ制限（100MB）に近づいている。コミットを中断し、対処法を提示する |

よくある大きなファイルの例：
- ビルド成果物（`.jar`, `.war`, `.exe`, `.dll`）
- データファイル（`.csv`, `.sql`, `.sqlite`, `.db`）
- メディアファイル（`.mp4`, `.mov`, `.psd`, `.ai`）
- パッケージ（`node_modules/`, `.venv/`, `vendor/`）

これらの拡張子やディレクトリが含まれている場合は、サイズに関わらず「本当にGit管理したいファイルか」を確認する。

## コミット時のワークフロー

コミットを求められたら、以下の手順で進める：

1. **ブランチ確認** — `git branch --show-current` で現在のブランチを確認。保護ブランチなら警告する
2. **変更内容の確認** — `git status` と `git diff` で変更を把握する
3. **安全チェック** — 「コミット前の安全チェック」セクションに従い、機密情報の検出とファイルサイズ確認を実施する。問題があれば警告する
4. **イシュー番号の確認** — ブランチ名にイシュー番号が含まれていればそれを使う。なければユーザーに「関連するイシュー番号はありますか？」と確認する（なければ省略可）
5. **適切なprefixの選択** — 変更内容からprefixを判断する
6. **コミットメッセージの作成** — フォーマットに従って日本語でメッセージを生成する
7. **ユーザーに確認** — メッセージを提示し、問題なければコミットを実行する

## ブランチ作成時のワークフロー

新しいブランチの作成を求められたら：

1. ブランチの目的を確認する（機能追加？バグ修正？リリース準備？）
2. 適切なブランチ種別（feature / hotfix / release）を選ぶ
3. 命名規則に従ったブランチ名を提案する
4. 正しい分岐元から作成する（featureはdevelopから、hotfixはmainから）

## プルリクエスト

PRを作成する際は以下を意識する：

- PRのサイズは小さく保つ。レビューしやすい範囲に分割する
- PRタイトルにもprefixとイシュー番号を含める
- マージ後はブランチを削除する
